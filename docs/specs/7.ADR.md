## Часть 5: ADR (Architecture Decision Records)

### Что это

Короткий markdown-документ, фиксирующий одно архитектурное решение. Формат предложил Майкл Найгард в 2011, сейчас это стандарт в зрелых командах.

### Зачем в песочнице

Песочница — инструмент для принятия решений. Эксперимент даёт данные, ADR фиксирует вывод. Без ADR через полгода ты забудешь, почему выбрал zustand, какие альтернативы рассматривал и что пошло не так с effector.

ADR превращает песочницу из свалки экспериментов в накопленную базу знаний.

### Шаблон ADR (`docs/decisions/template.md`)

```markdown
# NNNN. Заголовок решения

**Дата:** YYYY-MM-DD
**Статус:** proposed | accepted | deprecated | superseded by ADR-XXXX
**Связанные эксперименты:** experiments/foo, experiments/bar

## Контекст

Какую проблему решаем, какие были ограничения, что заставило выбирать.

## Решение

Что выбрали, кратко и однозначно.

## Альтернативы

- Вариант A — почему не подошёл
- Вариант B — почему не подошёл
- Вариант C — почему не подошёл

## Последствия

- Положительные эффекты

* Отрицательные эффекты, технический долг, что придётся пересмотреть в будущем
```

### Конвенции

- Имя файла: `NNNN-kebab-case-title.md`, где NNNN — четырёхзначный порядковый номер
- Один ADR — одно решение. Не смешивай «выбрал стейт-менеджер» и «выбрал router» в одном файле
- ADR не редактируется после статуса `accepted`. Если решение пересмотрено — создаётся новый ADR со статусом `accepted` и пометкой `supersedes ADR-XXXX`. Старый получает статус `superseded by ADR-YYYY`. Это сохраняет историю
- Связь с экспериментами обязательна, если ADR родился из эксперимента. Это даёт обратную трассировку: «откуда это решение → какой эксперимент его породил → какие данные были получены»

### Индекс `docs/decisions/README.md`

Простая таблица всех ADR:

```markdown
# Architecture Decision Records

| #    | Заголовок                     | Статус   | Дата       |
| ---- | ----------------------------- | -------- | ---------- |
| 0001 | Monorepo with pnpm workspaces | accepted | 2026-04-11 |
| 0002 | Oxlint instead of ESLint      | accepted | 2026-04-11 |
| 0003 | Vite as default bundler       | accepted | 2026-04-11 |
```

Может генерироваться скриптом — см. ниже.

### CLI-скрипт `scripts/adr.ts`

```bash
pnpm run adr new "Выбор стейт-менеджера"
```

Логика:

1. Находит максимальный номер ADR в `docs/decisions/`
2. Создаёт файл `NNNN-vybor-stejt-menedzhera.md` (next number)
3. Копирует содержимое из `template.md`, подставляет заголовок и дату
4. Открывает файл в `$EDITOR` (опционально)

```bash
pnpm run adr index
```

Перегенерирует `docs/decisions/README.md` на основе всех ADR-файлов.

### Команда Claude Code

`.claude/commands/adr-create.md` — создаёт ADR на основе текущего контекста разговора. Например, после обсуждения «zustand vs reatom» можно сказать `/adr:create state-management`, и Claude:

1. Прочитает связанные эксперименты (если они существуют)
2. Сгенерирует черновик ADR с заполненными секциями: контекст из обсуждения, решение из выводов, альтернативы из эксперимента
3. Создаст файл через `pnpm run adr new`
4. Покажет результат для ручной правки

`.claude/commands/adr-list.md` — выводит все ADR с фильтром по статусу.
